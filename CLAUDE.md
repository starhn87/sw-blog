# CLAUDE.md — 코드 생성 및 수정 가이드라인

## 핵심 원칙

좋은 코드란 유지보수하기 쉬운 코드다.
성급한 추상화는 하지 않느니 못하다.

## 코드 작성 규칙

### 단순함을 우선한다
- 같은 코드가 3번 이상 반복될 때만 추상화를 고려한다
- 헬퍼 함수, 유틸리티, 래퍼는 한 번만 쓸 거라면 만들지 않는다
- "나중에 필요할 것 같다"는 이유로 코드를 추가하지 않는다
- 3줄짜리 인라인 코드가 추상화된 함수보다 나을 수 있다

### 변경 범위를 최소화한다
- 요청받은 것만 변경한다. 주변 코드를 "개선"하지 않는다
- 버그 수정에 리팩토링을 끼워넣지 않는다
- 기존 코드에 docstring, 타입 어노테이션, 주석을 임의로 추가하지 않는다
- 주석은 "왜(why)"를 설명할 때만 단다. "무엇(what)"은 코드가 말한다

### 에러 처리는 실제 경계에서만
- 외부 입력(사용자, API, DB)만 검증한다
- 내부 함수 간 호출에 방어적 코딩을 하지 않는다
- "혹시 모를" 에러 핸들링, fallback, feature flag를 넣지 않는다

### 삭제할 때는 깔끔하게
- 사용하지 않는 코드는 주석 처리가 아니라 삭제한다
- `// removed`, `_unused` 같은 흔적을 남기지 않는다
- 하위 호환 shim이나 re-export를 만들지 않는다

## 기술 스택 규칙

### Next.js 15 (App Router)
- Server Component를 기본으로 사용한다. "use client"는 인터랙션이 필요한 곳에만
- 데이터 페칭은 서버 컴포넌트에서, 상태 관리는 클라이언트 컴포넌트에서
- `generateStaticParams`로 정적 생성 가능한 페이지는 정적 생성한다

### TypeScript
- `any`를 사용하지 않는다. 정말 불가피하면 `unknown` + 타입 가드를 쓴다
- 불필요한 인터페이스를 만들지 않는다. 한 곳에서만 쓰이면 인라인 타입으로 충분하다
- 타입은 `src/types/`에 모으되, 컴포넌트 전용 타입은 해당 파일에 둔다

### Tailwind CSS
- 인라인 className으로 스타일링한다. CSS 파일을 별도로 만들지 않는다 (globals.css 제외)
- cn() 유틸리티로 조건부 클래스를 조합한다
- 매직 넘버 대신 Tailwind 스케일을 사용한다 (예: `p-4` O, `p-[17px]` X)

### Cloudflare 환경
- 모든 API 라우트에 `export const runtime = 'edge'`를 선언한다
- `fs` 모듈은 빌드 타임에만 사용한다 (런타임 사용 불가)
- Node.js 내장 모듈 대신 Web API를 사용한다
- D1 바인딩은 `getRequestContext()`로 접근한다

## 파일 구조 규칙
- 새 파일은 기존 디렉토리 구조를 따른다 (`REBUILD_PLAN.md` 참고)
- 컴포넌트 파일명은 PascalCase, 유틸/훅은 camelCase
- 한 파일이 200줄을 넘으면 분리를 고려한다 (강제는 아님)

## 커밋 & PR 규칙
- 커밋은 영어로, 변경의 "왜"를 담는다
- 한 커밋에 한 가지 변경만 담는다
- 작동하지 않는 코드를 커밋하지 않는다

## 응답 스타일
- 한국어로 소통한다. 기술 용어와 코드는 원어 그대로 사용한다
- 요약부터 말하고, 필요하면 상세 설명을 덧붙인다
- 작업 완료 후 불필요한 요약을 반복하지 않는다
