코드는 이제 AI가 대부분 구현을 하는 시대가 되었어요.

구현이 빨라진 만큼 저의 시간은 다른 데로 옮겨갔어요. **구현 시간이 줄어든 만큼 판단과 설계에 시간을 더 쓰게 됐어요.** AI에게 외주를 주지 않고 같이 협업하면서도 주도권을 잃지 않고, 밸런스를 어느 수준으로 어떻게 맞출 지 계속 고민을 하고 있어요. 이 글은 그 고민으로부터 시작했어요.

## 1. 빠른 건 이제 디폴트다

AI와 일하면서 생산성이 비약적으로 오른 건 확실해요. 새 기능 하나에 며칠씩 걸리던 게 몇 시간으로 줄었어요. 이 블로그를 만들 때도 처음 적용한 기술이 손가락으로 세지 못할 정도인데 큰 병목 없이 빠르게 구현해서 배포할 수 있었어요.

문제는 나중에 그 코드를 다시 건드려야 할 때예요. 낯선 영역을 AI가 빠르게 만들어주고 잘 동작하면 그걸 내 꺼라고 할 수 있을까요?. 이건 학습이 아니라 외주에 가까워요.

그리고 이 속도는 이제 누구나 낼 수 있어요. '코드를 빨리 짜는 능력'은 이제 큰 변별력을 발휘하지 못해요. 그러면 뭐가 남을까.. 이게 가장 고민이 되는 질문이에요.

## 2. AI가 짠 코드를 의심하는 일

AI가 짠 코드는 표면적으로 보면 군더더기가 없어요. 변수명도 그럴듯하고 패턴도 익숙해요. 그래서 더 위험해요. 그럴듯한데 미묘하게 틀린 코드가 제일 잡기 어렵거든요.

한번은 AI가 useEffect 의존성에 값을 하나 추가하라고 했어요. 맞는 말처럼 보였어요. ESLint도 똑같이 시키니까요.

```tsx
useEffect(() => {
  // "의존성에 query를 추가하세요" - ESLint도 그렇게 시킨다
  search(query).then(setResults)
}, [query])
```

그런데 query가 바뀔 때마다 effect가 다시 돌았어요. 이전 요청이 끝나기도 전에 다음 요청이 나갔고, 응답이 뒤늦게 도착하면서 옛 결과가 최신 화면을 덮었어요. 전형적인 race condition이에요. AI가 준 건 교과서적으로 맞는 답이었어요. 다만 그 답이 우리 컴포넌트의 생명주기 위에서 어떻게 도는지까지는 몰라요. 그건 코드 흐름을 직접 그려본 다음에야 보여요.

코드를 짜는 능력보다 읽고 의심하는 능력이 더 무거워졌어요. 그런데 의심만으로는 부족하더라고요. 의심한 걸 일하는 방식에 녹여야 했어요. 여기서부터가 요즘 제가 시간을 많이 쓰는 자리예요.

## 3. AI가 환각 없이 일하게 만드는 구조

AI한테는 묘한 비대칭이 있어요. 세상의 코드는 거의 다 봤는데 정작 우리 프로젝트는 처음 봐요. 그래서 맥락을 안 주면 그럴듯한 거짓말을 해요. 우리 컨벤션이 아니라 인터넷 평균값으로 답하거든요.

이걸 매번 말로 설명하면 사람마다 다르게 전달돼요. 그래서 회사에서는 컨텍스트를 문서로 고정했어요. AGENTS.md나 CONVENTION.md 같은 공식 문서에 팀이 합의한 규칙을 적어두고 기능을 만들기 전에 Tech Spec을 먼저 썼어요. 이렇게 해두면 단순 반복 작업은 MCP나 스킬로 에이전트에 맡겨도 결과가 들쭉날쭉하지 않아요. 위임이 되려면 먼저 잘 쓰인 문서가 있어야 하더라고요.

개인 프로젝트도 비슷해요. 이 블로그에는 CLAUDE.md를 두고 "같은 코드가 세 번 넘게 반복될 때만 추상화한다" 같은 규칙을 적어놨어요. 회사 문서가 여럿이 같은 출발선에 서기 위한 거라면 이건 혼자라도 며칠 뒤의 나와 AI가 헤매지 않게 하려는 거예요.

여기까지는 사람이 합의해서 만드는 하네스예요. 그런데 사람이 만든 건 사람이 관리해야 하고, 관리가 밀리면 바로 문제가 생겨요.

## 4. 컨텍스트를 싱크하는 법

문서를 만들어두면 끝일 것 같지만 아니에요. 코드는 계속 바뀌는데 문서는 그 자리에 멈춰 있어요. 그러면 문서가 거짓말을 하기 시작해요. AI는 그 낡은 문서를 믿고 환각을 내고요.

이 블로그에 챗봇을 붙이면서 이 문제를 직접 만났어요. 글에는 "SEO는 나중에"라고 써놨는데 코드에는 sitemap이랑 RSS, JSON-LD가 이미 다 들어가 있었어요. 챗봇한테 SEO를 물으면 글만 보고 "아직 안 했다"고 답하더라고요. 사람이 보기엔 사소한데 AI한테는 사실이 두 개인 셈이에요.

그래서 빌드할 때마다 코드베이스 현황을 자동으로 뽑게 했어요. 구현된 페이지랑 API, 켜져 있는 기능을 코드에서 직접 읽어 요약 파일로 만들어요. 사람이 손으로 적는 게 아니라서 안 썩어요. 챗봇한테는 글과 실제 구현이 다르면 이 현황을 먼저 믿으라고 일러뒀고요.

답변에 쓰는 글 본문도 통째로 넣지 않아요. 500단어 정도로 자르되 앞뒤를 50단어씩 겹쳐서 잘라요. 문장이 청크 경계에서 끊겨 맥락이 날아가는 걸 막으려고요. 답변을 만들 땐 기본 지침과 코드베이스 현황, 질문에 맞는 글 조각을 세 겹으로 쌓아 보내요. 매번 똑같이 들어가는 현황은 캐시해 두고 재사용해서 토큰도 아끼고요.

<Callout type="tip">

문서는 만드는 것보다 안 썩게 두는 게 어려워요. 그래서 사람이 직접 관리하는 문서는 최소한으로 줄이고 나머지는 빌드가 알아서 만들게 했어요.

</Callout>

글을 하나 고쳐서 올리면 검색 색인과 챗봇이 보는 데이터까지 자동으로 따라와요. 이쪽은 사람이 아니라 기계가 알아서 유지하는 하네스예요. 이 블로그의 챗봇과 검색을 어떻게 만들었는지는 따로 적어둔 글이 있어요.

## 5. 최종 승인은 사람이 해요

컨텍스트를 잘 줘도 전부를 맡기진 않아요. 어디까지 AI에 넘기고 어디서 사람이 멈출지, 그 경계를 정하는 게 실전에서는 제일 중요했어요.

코드 리뷰가 좋은 예예요. 네이밍이나 접근성, GraphQL Fragment 규칙 같은 반복적인 지적은 LLM한테 1차로 맡겼어요. 매번 똑같이 걸리는 컨벤션은 아예 커스텀 ESLint 룰로 만들어 패키지로 묶었고요.

```ts
// 사람이 매번 짚던 컨벤션을 룰로 옮긴다
rules: {
  'team/img-needs-alt': 'error',
  'team/fragment-name-matches-component': 'error',
}
```

그랬더니 컨벤션 관련 리뷰 피드백이 0건이 됐어요. 사람이 리뷰에서 봐야 할 건 따로 있거든요. "이 기능을 이 방식으로 푸는 게 맞나" 같은 판단이요.

AI가 초안을 만들고 사람이 승인하는 구조도 같은 생각이에요. 예전에 어드민에서 콘텐츠를 여러 나라 말로 등록할 때 같은 작업을 어권 수만큼 반복해야 했어요. 이걸 한 번만 입력하면 AI가 각 언어 초안을 만들고 담당자가 확인한 뒤 한꺼번에 등록하는 흐름으로 바꿨어요. **번역을 AI가 만들어도 게시 버튼은 사람이 누른다는 거예요.**

<Callout type="warning">

AI한테 리뷰를 통째로 맡기는 게 아니에요. 사람이 꼭 봐야 할 것만 남기려고 반복 작업을 LLM과 룰에 미리 맡기는 거예요. 둘은 비슷해 보여도 책임의 위치가 달라요.

</Callout>

## 6. 무엇을 뺄지 정하는 일

설계와 판단에서 제일 어려운 건 더하는 게 아니라 빼는 거예요. AI는 두면 자꾸 더해요. 만약을 대비한 분기를 넣고 예외 처리를 겹겹이 쌓아요. 무엇을 뺄지는 앞서 말한 비대칭 때문에 사람이 정할 수밖에 없어요. 무엇이 필요 없는지는 우리 프로젝트를 아는 쪽만 알거든요.

성능 작업이 딱 그랬어요. 기능을 더하는 일만큼 무엇을 빼야 빨라지는지 보는 데 시간을 썼어요. AI한테 느린 페이지를 고쳐달라고 하면 보통 뭔가를 더 얹어요. 캐싱을 한 겹 덧대거나 메모이제이션을 추가하는 식이에요. 그런데 그때 필요한 건 더하기가 아니라 빼기였어요. 인스턴스마다 캐시를 따로 들고 있느라 같은 요청이 중복으로 쌓이던 구조였거든요. 캐시 책임을 CDN으로 넘기고 SSR로 바꿨어요. 콘텐츠 성격에 따라 캐시 수명을 다르게 뒀어요. 있는 캐시를 먼저 응답하고 뒤에서 조용히 갱신하게 했고요. 무거운 페이지는 로딩과 반응성, 레이아웃 흔들림을 따로 뜯어봤어요. 결과적으로 프론트 서버를 열 대에서 두 대로 줄였어요. 더한 게 아니라 덜어내서 나온 숫자예요.

추상화도 그래요. AI는 비슷한 코드 세 개를 보면 하나로 합치자고 해요. 그런데 그 셋이 우연히 닮은 거라면 나중에 서로 다른 방향으로 갈라져요. 그때 억지로 묶어둔 추상화는 풀어내기가 더 힘들어요. 잘못된 추상화는 중복보다 비싸거든요. AI는 닮은 걸 보면 합치려 하고 거기서 멈출지를 정하는 건 사람이에요.

## 마무리

코드를 빨리 짜는 일은 변별력을 잃고 있어요. 그래서 요즘 보여주고 싶은 건 그게 아니에요. 무엇을 모르는지 아는 능력, 그리고 무엇을 AI에 맡기고 무엇을 끝까지 안 맡겼는지 그 경계를 설명할 수 있는 능력이에요.

맨 앞에서 외주 이야기를 꺼냈어요. 구현은 얼마든지 넘겨요. 그런데 왜 이렇게 짜였는지, 뭘 빼야 하는지, 어디서 멈춰야 하는지는 넘기지 않아요. 그 자리까지 넘기면 정말 아무것도 안 남거든요. **AI에게 외주를 주되 외주를 주면 안 되는 걸 아는 것.** 요즘 제가 일하는 방식은 거기에 다 걸려 있어요.

그래서 새 작업을 받으면 코드부터 열지 않아요. AI한테 뭘 줄지 어디까지 맡길지부터 정리해요. 이직을 준비하면서도 결국 보여줄 건 이 경계라고 생각하고요.
